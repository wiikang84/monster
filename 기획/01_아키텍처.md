# 01. 기술 아키텍처 — Tower Defense (Unity 6 / WebGL)

> 작성: 2026-06-23 · 대상: Unity 6000.0.58f1, WebGL, Kenney CC0 아트, $0, 솔로 개발(코드는 Claude가 파일로 작성)
> 라이브: https://wiitower.web.app · repo: wiikang84/monster · 프로젝트: `tower/game`
> 이 문서의 목적: **모놀리식 GameManager(285줄, 모든 책임 하드코딩)** 를 **데이터 기반·모듈식·다스테이지** 구조로 전환하기 위한 착수 가능한 설계.

---

## 0. 현재 상태 진단 (사실)

`GameManager.cs` 한 파일에 다음이 전부 섞여 있다:
- 상태머신(`State.Ready/InWave/Won/Lost`), 그리드(10×10 상수), 경로(corners 하드코딩), 설치슬롯(slotCells 하드코딩)
- 웨이브(`waves` 리스트, basic/fast 두 종류만), 경제(gold/lives/cost 상수), 입력(`Input.GetMouseButtonDown(0)` + Raycast), HUD(`OnGUI` IMGUI)
- 모델 인스턴스화(`Make`/`SpawnModel`), 카메라/조명 생성, 머티리얼 생성

`Enemy`/`Tower`/`Projectile`은 `GameManager.Instance`에 직접 결합. 적·타워·발사체 스탯이 코드에 매직넘버로 박혀 있음(`hp=35/60`, `range=cell*3.5f` 등).

**문제의 본질**: "콘텐츠(스테이지·타워·적·웨이브)"와 "엔진(이동·전투·경제)"이 분리되지 않아, 콘텐츠 추가 = 코드 수정 = 빌드 리스크. 다스테이지·난이도 기획팀 협업·튜토리얼·업그레이드·저장이 전부 불가능한 구조.

## 1. 시스템 분해도

각 시스템은 **단일 책임**을 갖고, 직접 호출 대신 **이벤트(GameEvents) + 서비스 조회(ServiceLocator)** 로 느슨하게 연결한다. (DI 프레임워크는 도입하지 않음 — 솔로/소규모라 정적 ServiceLocator로 충분.)

```
┌─────────────────────────────────────────────────────────────┐
│ Boot (씬 1개 + 부트스트랩)                                      │
│  - 서비스 등록, ContentDB 로드, 진행도 로드 → AppFSM 시작        │
└───────────────┬─────────────────────────────────────────────┘
                │
        ┌───────▼────────┐   AppFSM (앱 레벨 상태머신)
        │ Title          │   Title → StageSelect → Tutorial? → InGame → Result
        │ StageSelect    │
        │ InGame         │
        │ Result         │
        └───────┬────────┘
                │ (InGame 진입 시 StageDef 1개를 주입)
   ┌────────────▼───────────────────────────────────────────┐
   │ BattleController (전투 1판의 오케스트레이터, BattleFSM)    │
   │   Prepare → Building(웨이브 대기) → InWave → Cleared/Lost │
   └─┬─────┬─────┬──────┬──────┬──────┬──────┬──────┬─────────┘
     │     │     │      │      │      │      │      │
  MapSvc Spawn Tower Target Economy Input  UI    Audio
   /Path  /Wave  Mgr   /Combat Svc   Svc   Layer  Svc
```

### 1.1 GameFlow / 씬 · 상태머신
- **씬은 계속 1개**(현 BuildScript가 빈 씬 1개 + RuntimeInitialize 부트스트랩을 쓰는 방침 유지). 화면 전환은 씬 로드가 아니라 **AppFSM 상태 전환 + UI 패널 토글**로 처리 → WebGL 로딩 끊김 없음.
- **AppFSM** (앱 레벨): `Title / StageSelect / Tutorial / InGame / Result`. 각 상태는 `Enter/Exit/Tick`을 갖는 `IAppState`.
- **BattleFSM** (전투 한 판): `Prepare → Building → InWave → WaveCleared → (다음 웨이브 Building | StageCleared) | Defeat`. 현재 `State` enum이 이 자리로 이동·세분화된다.
- 책임: 상위 흐름만 제어. 전투 로직은 BattleController 이하에 위임.

### 1.2 맵 / 경로 시스템 (MapService + PathService)
- **MapService**: StageDef의 텍스트 그리드를 읽어 타일(바닥/길/설치가능/스폰/기지)을 런타임 생성, 셀↔월드 좌표 변환(`Center(c,r)` 이전), 셀 크기 측정(`MeasureCell` 이전) 담당.
- **PathService**: 그리드에서 **경로 자동 추출**(스폰 타일 S → 기지 타일 G까지 길 타일 `#`을 BFS로 추적해 waypoint 리스트 생성). 다중 경로 지원(여러 S/G 또는 분기 → `List<List<Vector3>>`). 현재 corners 하드코딩을 대체.
- 상호작용: BattleController가 StageDef를 MapService에 넘김 → MapService가 PathService로 경로 산출 → SpawnSystem이 경로를, TowerSystem이 설치슬롯을 구독.

### 1.3 스폰 / 웨이브 시스템 (WaveRunner)
- WaveDef 리스트를 코루틴/타이머로 순차 실행. 한 웨이브 = 여러 **스폰 그룹**(적종류 × 수량 × 간격 × 시작딜레이 × 경로인덱스).
- "다음 웨이브 자동/수동" 모드, 웨이브 간 휴식시간, 마지막 웨이브 종료 판정(`allSpawned && 적 0`)을 담당.
- EnemyFactory(풀 기반)로 적 생성, 적에게 EnemyDef + 경로 주입.

### 1.4 타워 시스템 (TowerSystem + Tower)
- 설치/판매/업그레이드 관리. 슬롯 점유, 비용 차감(Economy 경유), TowerDef 기반 인스턴스화.
- `Tower`는 TowerDef를 들고 자기 스탯(사거리/발사율/데미지/투사체)을 데이터에서 읽음. 업그레이드 = `def.Upgrades[level]` 적용.

### 1.5 타게팅 / 전투 (TargetingService + CombatService)
- **TargetingService**: 타워의 타깃 선택 정책(가장 가까움/가장 앞선/최대HP/최저HP). TowerDef에 `Targeting` 필드로 지정. 현재 "가장 가까움" 하드코딩을 정책화.
- **CombatService**: 데미지 적용, 데미지 타입(단일/스플래시/슬로우/DoT)을 EnemyDef의 저항/면역과 대조. Projectile은 풀에서 꺼내 타깃·데미지패킷 주입.

### 1.6 경제 (EconomyService)
- gold/lives/스코어 단일 소유. 변동은 메서드로만(`TrySpend`, `Reward`, `LoseLife`). 변동 시 `GameEvents.GoldChanged` 등 발행 → UI가 구독(폴링 제거).
- 시작 골드/라이프/처치 보상/타워 비용은 전부 StageDef·TowerDef·EnemyDef에서 읽음(상수 제거).

### 1.7 입력 추상화 (InputService) — 마우스 + 터치
- 플랫폼 분기를 한 곳에 격리. 외부엔 **의도(intent) 이벤트**만 노출:
  `TapWorld(Vector2 screenPos)`, `HoverWorld`, `Cancel`, `ConfirmPlacement`.
- 내부: 마우스(`Input.GetMouseButtonDown`)와 터치(`Input.touchCount`/`Touch.phase`)를 모두 처리, **UI 위 탭은 무시**(`EventSystem.IsPointerOverGameObject`). 멀티터치/핀치줌은 후순위 훅만 남김.
- Raycast로 슬롯/타워 히트 판정 후 의도 이벤트 발행 → TowerSystem이 구독. (현재 Update 내 직접 Raycast+설치를 분리)

### 1.8 UI 레이어 (uGUI Canvas) — **IMGUI 졸업**
- 화면별 패널: TitlePanel / StageSelectPanel / HUDPanel / TowerActionPanel(설치·업그레이드·판매) / TutorialPanel / ResultPanel.
- UI는 **GameEvents 구독으로만** 데이터를 받음(게임 로직을 호출하지 않고, 버튼 → intent 이벤트 발행). UX팀과의 계약면.
- 왜 IMGUI를 버리나: 아래 §3 참조.

### 1.9 오디오 (AudioService)
- BGM 1채널 + SFX 풀(AudioSource 여러 개 회전). 키 기반 재생: `Play(SoundId.TowerShoot)`.
- 사운드 매핑은 데이터(ScriptableObject `AudioBank` 또는 enum→경로 딕셔너리). 이미 `Assets/Audio`에 Kenney 7팩 보유(laser*, impact*, ui* 등) — 코드가 직접 OGG 경로를 알지 않게 Bank로 격리.
- **WebGL 주의**: 브라우저 자동재생 정책 → 첫 사용자 입력(타이틀 탭) 후에야 AudioContext 시작. BGM은 첫 탭 이후 재생.

### 1.10 저장 / 진행도 (SaveService + ProgressService)
- WebGL 저장 = `PlayerPrefs`(IndexedDB로 영속) 또는 직접 JSON을 PlayerPrefs 문자열로. 솔로/소규모엔 PlayerPrefs+JSON으로 충분(서버 0원).
- **ProgressService**: 스테이지 해금(직전 클리어 시 다음 해금), 별점(1~3, 클리어 시 남은 라이프 비율로 산정), 최고기록, 설정(음량). 데이터: `SaveData { unlockedStageIndex, Dictionary<stageId, stars>, settings }`.
- StageSelect가 ProgressService를 읽어 잠금/별 표시. 전투 종료 시 Result → ProgressService.Record → Save.

## 2. 데이터 기반 콘텐츠 스키마

### 2.1 형식 권고 — **ScriptableObject + 빌드시 JSON 임포트 하이브리드**

| 후보 | 장점 | 단점 |
|---|---|---|
| **ScriptableObject(SO)** | Unity 네이티브, 인스펙터 편집, 참조(에셋/프리팹) 직접 연결, WebGL 정상 | 텍스트 diff/머지 불리, 코드(Claude)로 대량 생성 불편(에셋 GUID 필요) |
| **JSON(Resources/StreamingAssets)** | **Claude가 파일로 직접 작성·diff 쉬움**, 레벨팀이 텍스트로 채움, git 친화 | 에셋 참조는 문자열 키로 우회 필요, 스키마 검증 직접 구현 |

**결론(하이브리드, code-first 정신 유지):**
- 콘텐츠 정의(TowerDef/EnemyDef/WaveDef/StageDef)는 **JSON을 1차 진실(source of truth)** 로 한다. `Assets/Resources/Content/*.json` → `Resources.Load<TextAsset>` → `JsonUtility`/Newtonsoft로 역직렬화해 런타임 SO 인스턴스(`ScriptableObject.CreateInstance`)로 적재 → `ContentDB`가 id→def 딕셔너리로 보관.
- **근거**: ① 코드는 Claude가 파일로 작성하는 워크플로우 → 텍스트 JSON이 SO 에셋보다 작성·리뷰·diff가 압도적으로 쉬움. ② 레벨/난이도 기획팀이 Unity 없이도 텍스트로 스테이지를 채울 수 있음(협업 계약면이 깔끔). ③ "모델 이름/사운드 키"는 이미 문자열(`Resources.Load("Models/"+name)`)이라 JSON과 궁합이 맞음. ④ WebGL에서 `Resources` + `JsonUtility` 모두 정상.
- **에셋 참조**: 직접 오브젝트 참조 대신 **문자열 키**(`"enemy-ufo-a"`, `"laser1"`)로 두고 로더가 `Resources.Load`로 해소(현 방식과 동일) → JSON 친화 유지.
- SO를 완전히 버리진 않음: 향후 인스펙터 편집이 더 편한 전역 설정(난이도 곡선, 오디오뱅크)은 SO로 둘 수 있음. 콘텐츠 대량 데이터만 JSON.

`JsonUtility`는 다형성/딕셔너리에 약하므로, **Newtonsoft.Json(com.unity.nuget.newtonsoft-json, 무료/UPM)** 도입 권고. WebGL 정상 동작.

### 2.2 스테이지 맵 = 텍스트 그리드 (하드코딩 제거 핵심)

좌표 corners/slotCells 하드코딩 대신 **ASCII 그리드**로 맵을 표현한다. 레벨팀이 눈으로 그리고 텍스트로 채운다.

```
범례:  . 빈 바닥(설치 불가)   # 길(적 이동)   o 설치 가능 슬롯
       S 스폰점(길)          G 기지/도착점(길)
```
행은 위(북, r 큰 값)부터 아래로 적는다(에디터에서 보기 편하게). 로더가 행 순서를 뒤집어 (c,r)로 매핑.

PathService가 `S`에서 `#`/`G`를 따라 BFS로 waypoint를 자동 생성 → 경로 좌표를 손으로 적을 필요 없음. 다중 `S`/`G`면 다중 경로.

## 3. code-first 유지 vs 프리팹/SO/Canvas — 권고 (실용 하이브리드)

| 영역 | 권고 | 근거 |
|---|---|---|
| 콘텐츠 데이터 | **JSON(런타임 SO로 적재)** | §2.1. Claude/레벨팀 텍스트 편집·diff·검증 용이 |
| 맵/적/타워/발사체 생성 | **code-first 유지** | 현 `Resources.Load + Instantiate` 방식 좋음. 단 풀링 추가(§6) |
| UI | **uGUI Canvas로 전환(IMGUI 폐기)** | IMGUI(OnGUI)는 매 프레임 재구성·레이아웃 수동·터치/DPI 대응 빈약·애니메이션 불가·디자인 한계. 사용자 피드백("뭐가 뭔지 모르겠다")의 직접 원인. uGUI는 WebGL 정상, 패널/버튼/툴팁/튜토리얼 오버레이에 적합. **단, Canvas는 프리팹 없이 코드로도 구성 가능**하나, 화면 수가 많아지므로 화면당 1프리팹 권고(편집 효율) |
| 전역 설정(난이도곡선/오디오뱅크) | SO 선택적 | 인스펙터 편집이 더 편한 소수 데이터만 |
| Addressables | **도입 안 함(현 단계)** | 콘텐츠 규모 작음, Resources로 충분. 모델 수백 개로 커지면 그때 |

요지: **"엔진 로직과 오브젝트 생성은 code-first 유지, 콘텐츠는 JSON 데이터화, UI는 uGUI로 졸업"** 의 3분할.

## 4. 마이그레이션 계획 (항상 빌드·플레이 가능)

각 단계는 끝나면 즉시 Play/WebGL 빌드가 동작해야 한다. GameManager를 한 번에 부수지 않고 **새 시스템을 옆에 세우고 호출을 옮긴 뒤 옛 코드를 주석화**(프로젝트 규칙: 삭제 말고 주석)한다.

- **M0. 안전망**: 현재 빌드 동작 확인 + git 커밋(롤백 지점). 폴더/네임스페이스 골격(§5)과 빈 ServiceLocator/GameEvents 추가. (게임 동작 변화 없음)
- **M1. 데이터 추출**: ContentDB + JSON 로더 추가. towers/enemies/stage_01 JSON 작성(현 수치 그대로). 단 아직 GameManager는 기존 하드코딩으로 동작 — JSON 로드만 검증(로그). **동작 동일**.
- **M2. 맵/경로 데이터화**: MapService+PathService 도입, stage_01.grid에서 타일·경로·슬롯 생성. GameManager의 `BuildPathData/BuildEnvironment/corners/slotCells` 주석화. 외형/동작 동일하게 보이도록 grid를 현 좌표와 맞춤.
- **M3. 적/타워/전투 데이터화**: EnemyFactory/TowerSystem이 Def에서 스탯을 읽도록. Enemy/Tower/Projectile의 매직넘버 제거, Def 주입. 데미지타입/타게팅 정책 도입(우선 Single/Nearest만 채우고 나머지 스텁).
- **M4. 경제+이벤트화**: EconomyService 도입, gold/lives 이동, 변동 이벤트 발행. (UI는 아직 IMGUI지만 이벤트값을 읽도록)
- **M5. UI uGUI 전환**: HUDPanel/TowerActionPanel/ResultPanel을 Canvas로. `OnGUI` 주석화. 입력도 InputService로 분리. **여기서 "사용법 모르겠다" 해소(설치 안내·툴팁·웨이브 표시)**.
- **M6. 다스테이지 + 진행도**: AppFSM(Title/StageSelect/Result) + ProgressService(해금/별점/저장). stage_02~ 추가. StageSelect 화면.
- **M7. 튜토리얼**: TutorialPanel + tutorialId 스텝(첫 스테이지에서 "여기 탭해 설치" 등 단계 안내).
- **M8. 업그레이드 + 오디오 + 폴링제거 마감**: 타워 업그레이드/판매 UI, AudioService+Bank, 풀링(§6) 적용, 잔여 정리.

각 M 종료 시 WebGL 빌드 1회 + wiitower 배포로 회귀 확인.

## 6. 오브젝트 풀링 · WebGL 성능 · 빌드 파이프라인

### 6.1 풀링
- 적·발사체는 현재 매번 `Instantiate`/`Destroy` → WebGL에서 GC 스파이크·프레임 드랍 위험. **EnemyFactory/ProjectileSystem이 `ObjectPool<T>`** 로 재사용(비활성화 후 풀 복귀).
- 풀 항목은 Def별로 분리(모델이 다름). 라운드 시작 시 예열(prewarm) 소량.
- `Tower.Update`의 매 프레임 전체 적 순회(O(타워×적))는 적·타워가 많아지면 비용↑ → 후순위로 **공간 분할(그리드 버킷)** 또는 타워별 타깃 캐시(N프레임마다 재탐색)로 완화. MVP 규모(수십)에선 현 방식 유지.

### 6.2 WebGL 성능 주의
- **GC 최소화**: 매 프레임 `new`(Vector 리스트/델리게이트 클로저) 금지. 이벤트 발행도 박싱 피함.
- **머티리얼**: 현재 `sharedMaterial`로 통일(좋음). 인스턴싱 켜기. EMISSION 슬롯 강조는 머티리얼 1개 공유.
- **오디오**: WebGL은 압축 디코딩 비용 큼 → SFX는 짧은 OGG, AudioSource 풀로 동시재생 제한. 자동재생 정책(첫 입력 후 시작).
- **빌드 용량**: Resources에 넣은 모든 모델이 빌드에 포함됨. 미사용 FBX는 Resources 밖으로. (현 13개는 OK)
- **메모리**: WebGL 기본 힙 제한 주의. 텍스처 압축(colormap 등) 유지.

### 6.3 빌드 파이프라인 영향
- `BuildScript.BuildWebGL`은 거의 그대로 유효(빈 씬 + 부트스트랩). **단 Bootstrap 진입점이 Core/Bootstrap.cs로 이동**하므로 RuntimeInitialize 위치만 변경.
- Newtonsoft 추가 시 UPM 패키지 의존성 → `Packages/manifest.json`에 `com.unity.nuget.newtonsoft-json` 추가 필요(빌드 전 1회).
- 콘텐츠가 `Resources/Content/*.json`이면 빌드에 자동 포함(별도 조치 불필요).
- `IL2CPP + WebGL`에서 리플렉션 기반 역직렬화는 코드 스트리핑 주의 → `link.xml`로 Def 타입 보존하거나 Newtonsoft의 명시적 설정 사용.

## 7. 다른 팀과의 계약 / 의존성 / 열린 질문

### 7.1 레벨·난이도 기획팀
- **소유물**: `Assets/Resources/Content/*.json` 의 StageDef/WaveDef(별점 컷 포함). §2.2 그리드 규칙 + §2.3 스키마가 계약.
- **보장(엔진이 제공)**: 그리드만 그리면 경로·타일·슬롯 자동 생성(PathService.Validate가 오류를 빌드/로드시 경고), 사용 가능한 towers/enemies id 목록, 별점 자동 산정.
- **제약**: 새 타워/적 "종류"가 필요하면 towers/enemies.json에 추가(데이터) — 단 새 **메커니즘**(예: 공중 적, 분기 경로 신규)이면 엔진 작업(Claude) 필요.

### 7.2 UX / UI 팀
- **구독 계약**: UI는 `GameEvents`(GoldChanged/LivesChanged/WaveChanged/BattleEnded/SlotSelected)만 구독해 표시. 게임 로직을 직접 호출하지 않음.
- **발행 계약**: 버튼/탭은 intent 이벤트(설치요청/업그레이드요청/판매요청/웨이브시작)만 발행. 어떤 타워를 지을지는 `allowedTowers` + TowerDef(displayName/cost)에서 읽음.
- **튜토리얼**: TutorialPanel은 tutorialId 스텝 데이터(타깃 UI/문구)를 받음 — 스텝 데이터 스키마는 M7에서 확정.
- 사용자 핵심 피드백("사용법·뭐가 뭔지 모름") 해소 책임은 M5(HUD/툴팁/설치 안내)+M7(튜토리얼)에 있음.

### 7.3 결정이 필요한 열린 질문
1. **역직렬화 라이브러리**: Newtonsoft(다형성/딕셔너리 강함, 권고) vs JsonUtility(의존성 0이나 제약). → 권고: Newtonsoft.
2. **웨이브 진행**: 수동(시작 버튼) vs 자동 vs 스테이지별 선택(autoNextWave). 초보 친화는 수동, 긴장감은 자동.
3. **별점 기준**: 남은 라이프 비율로 충분한가, 아니면 클리어 시간/노데미지 보너스도?
4. **저장 범위**: 진행도만 vs 진행 중 세이브(전투 중간 저장은 WebGL 새로고침 대비 가치 있으나 복잡). → 권고: 진행도만.
5. **다중 경로/공중 적**을 MVP+1에 넣을지(엔진 작업량 큼) — 레벨팀 수요에 달림.
6. **타워 배치 방식**: 고정 슬롯(현재) vs 자유 그리드 설치(길 막기). 게임성·난이도 설계에 큰 영향 → 기획팀 결정 필요.
7. **모델/사운드 키 네이밍 규칙** 확정(레벨팀이 JSON에 적을 문자열) — Kenney 원본 파일명 기준으로 고정 권고.
